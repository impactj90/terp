
**Key observations:**
- No `deletedAt` field -- the Booking model does NOT support soft-delete at the schema level
- The Go `Delete` is also a hard delete (GORM Delete with RowsAffected check)
- The ticket says "Soft-Delete" for the delete mutation, but the schema has no soft-delete column. This is a discrepancy that needs clarification. The implementation should use hard delete (matching existing behavior) unless a migration adds a `deletedAt` column.
- Time values stored as minutes from midnight (0-1439 integer), not HH:MM strings
- `source` values: "web", "terminal", "api", "import", "correction", "derived"

## 4. Middleware Patterns

File: `/home/tolga/projects/terp/apps/web/src/server/middleware/authorization.ts`

### 4.1 `requirePermission(...permissionIds: string[])`
Checks if user has ANY of the specified permissions (OR logic). Used for simple permission gates.

### 4.2 `requireSelfOrPermission(userIdGetter, permissionId)`
Allows if target user ID matches current user OR if user has the fallback permission. Takes a `(input: unknown) => string` getter function.

### 4.3 `requireEmployeePermission(employeeIdGetter, ownPermission, allPermission)`
Handles "own vs all" employee-scoped access:
- If user's employeeId matches target: requires ownPermission OR allPermission
- If different employee: requires allPermission only
- Admin bypass built in
- Takes a `(input: unknown) => string` getter function

### 4.4 `applyDataScope()`
Adds DataScope object to context based on user's data scope configuration. Used by routers to filter Prisma queries.

## 8. Pair Booking Logic

The Go repository has pair management methods:
- `SetPair(bookingID1, bookingID2)` - generates a shared UUID pair_id and sets it on both bookings in a transaction
- `ClearPair(pairID)` - removes pair_id from all bookings sharing that pairID
- `GetUnpaired(tenantID, employeeID, date, direction)` - finds unpaired bookings of a specific direction

**How pairing works:** The `pair_id` field is a UUID that links two bookings together (typically an "in" and "out" booking). Both bookings in a pair share the same `pair_id`. Pairing is done externally (calculation engine pairs bookings), not during CRUD operations.

**For TICKET-232:** The CRUD router does NOT need to implement pair management. Pairing is handled by the recalculation service (TICKET-235). The pair_id is just a readable field on bookings.

## 10. Data Scope Handling

The employees router demonstrates the full data scope pattern:
1. Apply `applyDataScope()` middleware to get scope in context
2. For list queries: use `buildDataScopeWhere(dataScope)` to add WHERE conditions

## 12. Important Discrepancies Between Ticket and Codebase

1. **Permissions**: The ticket references `bookings.read_own`, `bookings.read`, `bookings.write_own`, `bookings.write` which do not exist in the permission catalog. The Go implementation uses `time_tracking.view_own`, `time_tracking.view_all`, `time_tracking.edit`, and `booking_overview.delete_bookings`.

2. **Soft-Delete**: The ticket specifies soft-delete for the delete mutation, but the Prisma Booking model has no `deletedAt` field, and the Go implementation uses hard delete. Implementing soft-delete would require a schema migration.

3. **Input Fields**: The ticket's create input mentions `time?, end_time?, duration_minutes?` which don't match the Go model or Prisma schema. The existing model uses `originalTime` and `editedTime` (minutes from midnight, integer). There is no `end_time` or `duration_minutes` field on the Booking model.

4. **Recalc Trigger**: The ticket explicitly excludes recalculation trigger (deferred to TICKET-235). The Go service calls `TriggerRecalc` after every mutation. The new tRPC router should NOT call recalc.

5. **Source field**: The ticket mentions `source?` in the create input. The Go handler hardcodes `BookingSourceWeb` for web-created bookings.
