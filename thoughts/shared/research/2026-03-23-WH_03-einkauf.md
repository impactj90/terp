## 2. Architecture Patterns

### Router Pattern

```ts
// Permission constants at top
const WH_VIEW = permissionIdByKey("wh_articles.view")!
const WH_CREATE = permissionIdByKey("wh_articles.create")!

// Base procedure with module guard
const whProcedure = tenantProcedure.use(requireModule("warehouse"))

// Each procedure chains: whProcedure -> requirePermission -> input validation -> service call -> handleServiceError
export const whArticlesRouter = createTRPCRouter({
  list: whProcedure
    .use(requirePermission(WH_VIEW))
    .input(z.object({ ... }))
    .query(async ({ ctx, input }) => {
      try {
        return await whArticleService.list(
          ctx.prisma as unknown as PrismaClient,
          ctx.tenantId!,
          input
        )
      } catch (err) {
        handleServiceError(err)
      }
    }),
})
```

### Repository Pattern

```ts
import type { PrismaClient } from "@/generated/prisma/client"
import { Prisma } from "@/generated/prisma/client"
import { tenantScopedUpdate } from "@/lib/services/prisma-helpers"

// Queries always include tenantId in where clause
export async function findById(prisma, tenantId, id) {
  return prisma.whArticle.findFirst({ where: { id, tenantId }, include: { ... } })
}

// Updates use tenantScopedUpdate helper (updateMany + refetch)
export async function update(prisma, tenantId, id, data) {
  return tenantScopedUpdate(prisma.whArticle, { id, tenantId }, data, { entity: "WhArticle" })
}

// Sub-entity updates verify tenant via parent relation first
export async function updateSupplier(prisma, tenantId, id, data) {
  const existing = await prisma.whArticleSupplier.findFirst({
    where: { id, article: { tenantId } },
  })
  if (!existing) return null
  return prisma.whArticleSupplier.update({ where: { id }, data })
}
```

### NumberSequence Pattern

File: `src/lib/services/number-sequence-service.ts`

```ts
// Default prefixes for auto-created sequences
const DEFAULT_PREFIXES: Record<string, string> = {
  customer: "K-",
  supplier: "L-",
  article: "ART-",
  // ... etc
}

export async function getNextNumber(prisma, tenantId, key): Promise<string> {
  const seq = await prisma.numberSequence.upsert({
    where: { tenantId_key: { tenantId, key } },
    update: { nextValue: { increment: 1 } },
    create: { tenantId, key, prefix: defaultPrefix, nextValue: 2 },
  })
  const value = seq.nextValue - 1
  return `${seq.prefix}${value}`
}
```

## 8. Database Migration Pattern

### Latest Migration Number

Latest: `20260323100000_wh_articles_artikelstamm.sql`

New migration should follow format: `20260323HHMMSS_wh_purchase_orders.sql` (e.g., `20260323110000`)

### SQL Migration Structure

From `20260323100000_wh_articles_artikelstamm.sql`:

```sql
-- 1. Create tables
CREATE TABLE wh_articles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  -- ... columns
  CONSTRAINT uq_wh_articles_tenant_number UNIQUE(tenant_id, number)
);

-- 2. Create indexes
CREATE INDEX idx_wh_articles_tenant_group ON wh_articles(tenant_id, group_id);

-- 3. Junction tables reference parent with CASCADE
CREATE TABLE wh_article_suppliers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  article_id UUID NOT NULL REFERENCES wh_articles(id) ON DELETE CASCADE,
  -- ...
);
```

**Important pattern:** Column names use `snake_case`, Prisma model uses `camelCase` with `@map()`.

### Enums in SQL

For the new enums (`WhPurchaseOrderStatus`, `WhPurchaseOrderMethod`), create as PostgreSQL types:

### Reorder Suggestions Query Logic

The `belowMinStock` filter in the article repository (lines 36-88 of `wh-article-repository.ts`) shows the challenge of field-to-field comparison in Prisma. It uses in-memory filtering:

```ts
if (params.belowMinStock) {
  where.stockTracking = true
  where.minStock = { not: null }
  // Then filter in-memory: a.minStock !== null && a.currentStock < a.minStock
}
```

The reorder suggestions query will need a similar approach, but also needs to:
1. Join to WhArticleSupplier to find supplier details
2. Calculate `suggestedQty = max(minStock - currentStock, defaultOrderQty)`
3. Optionally filter by supplierId

### Position Auto-Fill Logic

When adding a position to a PO for a specific article:
1. Look up `WhArticleSupplier` where `articleId` matches AND `supplierId` matches the PO's supplier
2. Auto-fill: `supplierArticleNumber`, `buyPrice` (as `unitPrice`), `orderUnit` (as `unit`)
3. If no supplier link found, fall back to article's `buyPrice` and `unit`

### Total Calculation

When positions are added/updated/deleted, recalculate:
```ts
subtotalNet = sum(position.totalPrice for all positions)
totalGross = subtotalNet  // No VAT calculation on purchase orders per ticket
```

Position `totalPrice = (quantity * unitPrice) + (flatCosts ?? 0)`

### Inquiry Link

The `inquiryId` field links to `CrmInquiry`. This is optional and follows the same pattern as billing documents linking to inquiries.
