## Date: 2026-03-21

## Booking Model

### Current Schema & Indexes

```prisma
model Booking {
  id            String   @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  tenantId      String   @map("tenant_id") @db.Uuid
  employeeId    String   @map("employee_id") @db.Uuid
  bookingDate   DateTime @map("booking_date") @db.Date
  bookingTypeId String   @map("booking_type_id") @db.Uuid

  // Time values (minutes from midnight, 0-1439)
  originalTime   Int  @map("original_time") @db.Integer
  editedTime     Int  @map("edited_time") @db.Integer
  calculatedTime Int? @map("calculated_time") @db.Integer

  // Pairing
  pairId String? @map("pair_id") @db.Uuid

  // Metadata
  source     String? @default("web") @map("source") @db.VarChar(20)
  terminalId String? @map("terminal_id") @db.Uuid
  notes      String? @db.Text

  // Booking reason and derived booking fields (migration 000078)
  bookingReasonId   String? @map("booking_reason_id") @db.Uuid
  isAutoGenerated   Boolean @default(false) @map("is_auto_generated")
  originalBookingId String? @map("original_booking_id") @db.Uuid

  // Timestamps and audit
  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt DateTime @default(now()) @map("updated_at") @db.Timestamptz(6)
  createdBy String?  @map("created_by") @db.Uuid
  updatedBy String?  @map("updated_by") @db.Uuid

  // ... relations omitted ...

  // Indexes
  @@index([tenantId], map: "idx_bookings_tenant")
  @@index([employeeId, bookingDate], map: "idx_bookings_employee_date")
  @@index([bookingDate], map: "idx_bookings_date")
  @@map("bookings")
}
```

**Existing indexes (from migration 000023 + 000079):**

| Index Name | Columns | Notes |
|---|---|---|
| `idx_bookings_tenant` | `(tenant_id)` | Tenant-only filter |
| `idx_bookings_employee_date` | `(employee_id, booking_date)` | Missing `tenant_id` prefix |
| `idx_bookings_date` | `(booking_date)` | Date-only filter |
| `idx_bookings_pair` | `(pair_id) WHERE pair_id IS NOT NULL` | Partial index |
| `idx_bookings_reason` | `(booking_reason_id) WHERE ...` | Partial index |
| `idx_bookings_auto_gen` | `(is_auto_generated) WHERE ...` | Partial index |
| `idx_bookings_original` | `(original_booking_id) WHERE ...` | Partial index |

**Problem:** The most common query pattern is `WHERE tenant_id = X AND employee_id = Y AND booking_date = Z` (or a date range). The current `idx_bookings_employee_date` index on `(employee_id, booking_date)` does NOT include `tenant_id`, meaning PostgreSQL must either:
- Use `idx_bookings_tenant` (tenant_id) and do a heap scan + filter on employee_id/bookingDate, or
- Use `idx_bookings_employee_date` (employee_id, booking_date) and do a recheck filter on tenant_id.

Neither is optimal for the hot path.

### Query Patterns Found

```ts
this.prisma.booking.findMany({
```

```ts
  where: { tenantId, employeeId, bookingDate: { gte: extFrom, lte: extTo } },
  ...
})
```

**4. `employees-repository.ts:192-193` — Employee day detail view**
```ts
prisma.booking.findMany({
  where: { tenantId, employeeId, bookingDate: date },
  ...
})
```

**5. `bookings-repository.ts:59-69` — List bookings (buildListWhere)**

```ts
prisma.booking.findMany({ where, ... })
```

**6. `evaluations-repository.ts:130-177` — Evaluation queries**
```ts

prisma.booking.findMany({ where, ... })
```

**7. `daily-calc.ts:660-668` — Concurrent creation guard**
```ts

  where: { tenantId, employeeId, bookingDate: date, bookingTypeId, source, notes, editedTime },
  ...
})
```

All 7 query patterns filter by `tenantId + employeeId + bookingDate` — the proposed composite index `(tenantId, employeeId, bookingDate)` covers all of them.

## MonthlyValue Model

### Current Schema & Indexes

```prisma

  // ... relations omitted ...

  // Indexes
  @@unique([employeeId, year, month])
  @@index([tenantId], map: "idx_mv_tenant")
  @@index([employeeId], map: "idx_mv_employee")
  @@map("monthly_values")
}
```

**Existing indexes (from migration 000029):**

| Index Name | Columns | Notes |
|---|---|---|
| `monthly_values_employee_id_year_month_key` | `(employee_id, year, month)` UNIQUE | Prisma @@unique — used by `findUnique` |
| `idx_monthly_values_tenant` / `idx_mv_tenant` | `(tenant_id)` | Tenant-only filter |
| `idx_monthly_values_employee` / `idx_mv_employee` | `(employee_id)` | Employee-only filter |
| `idx_monthly_values_lookup` | `(employee_id, year, month)` | Redundant with UNIQUE constraint |
| `idx_monthly_values_period` | `(year, month)` | Period-only filter, no tenant |

**Problem:** Queries that filter by `tenant_id + year + month` (the payroll export and monthly listing pattern) must use `idx_mv_tenant` (tenant_id only) and then do a heap scan + filter on year/month. The existing `idx_monthly_values_period` on `(year, month)` lacks `tenant_id`, so a BitmapAnd of both indexes is the best PostgreSQL can do — much worse than a single composite index.

### Query Patterns Found

**1. `monthly-values-repository.ts:42-46` — List monthly values for a period (primary listing endpoint)**
```ts

prisma.monthlyValue.findMany({ where, ... })
prisma.monthlyValue.count({ where })
```
This is the main monthly evaluation view: all employees for a given tenant+year+month.

**2. `payroll-export-repository.ts:163-164` — Batch fetch for payroll export**
```ts

  where: { tenantId, employeeId: { in: employeeIds }, year, month },
})
```
Payroll export fetches all monthly values for a batch of employees in a given period. The `tenantId + year + month` prefix narrows the scan before the `IN` filter.

**3. `payroll-export-repository.ts:150-152` — Single employee lookup for payroll**
```ts
prisma.monthlyValue.findFirst({
  where: { tenantId, employeeId, year, month },
})
```

**4. `monthly-values-repository.ts:123-124` — Find by employee+month (with tenant guard)**
```ts
prisma.monthlyValue.findFirst({
  where: { employeeId, year, month, tenantId },
})
```

**5. `monthly-values-service.ts:325-326` — Batch fetch during close/reopen**
```ts

  where: { employeeId: { in: employeeIds }, year, month },

```
(Note: this query is missing `tenantId` but still benefits from the proposed index since `year + month` reduces the scan set.)

**6. `monthly-calc.ts:325-327` — Atomic upsert during monthly calculation**

## Migration Convention

**Naming pattern:** `20260101NNNNNN_snake_case_description.sql`
- All migrations use timestamp prefix `20260101` followed by a 6-digit sequence number (zero-padded)
- Latest migration: `20260101000107_add_macro_assignment_last_executed.sql`
- Next migration number: `000108`
- Descriptions are snake_case, typically starting with a verb: `create_`, `add_`, `alter_`, `rename_`, `seed_`
- Index naming convention: `idx_<table>_<column_shorthand>` (e.g., `idx_bookings_employee_date`, `idx_mv_tenant`)

**Migration structure:**
- Plain SQL (not Prisma migrate)
- Comment header with migration number and description
- `CREATE INDEX` statements with explicit index names
- Index names follow `idx_<table_name>_<descriptive_suffix>` pattern

## Findings Summary

1. **Booking `@@index([tenantId, employeeId, bookingDate])`** — Strongly justified. Found **7 distinct query patterns** in the codebase that filter on `tenantId + employeeId + bookingDate` (exact match or date range). The daily calculation engine (`daily-calc.ts`) is the primary consumer and calls this pattern for every employee/day. The existing `idx_bookings_employee_date` on `(employee_id, booking_date)` forces PostgreSQL to either skip the tenant filter or merge with `idx_bookings_tenant` via BitmapAnd. A composite `(tenant_id, employee_id, booking_date)` index is the correct covering index for this workload.

2. **MonthlyValue `@@index([tenantId, year, month])`** — Strongly justified. Found **6 query patterns** that filter by `tenantId + year + month`. The monthly evaluation list and payroll export are the primary consumers, both scanning all employees for a given tenant+period. The existing `idx_mv_tenant` (tenant_id only) and `idx_monthly_values_period` (year, month without tenant_id) require PostgreSQL to do an index intersection. A composite `(tenant_id, year, month)` index directly satisfies these queries.

3. **Migration:** Next migration should be `20260101000108_add_composite_indexes_bookings_monthly_values.sql`. The Prisma schema also needs `@@index` directives added to both models.

4. **No existing index conflicts:** The new indexes do not duplicate existing ones. The existing `idx_bookings_employee_date` on `(employee_id, booking_date)` remains useful for queries that don't filter by tenant (none found, but kept for safety). The existing `idx_monthly_values_period` on `(year, month)` similarly remains but is largely superseded by the new composite.
