### Key Discoveries

- **Permissions**: The ticket references `bookings.read_own`, `bookings.read`, `bookings.write_own`, `bookings.write` which do NOT exist in the permission catalog (`apps/web/src/server/lib/permission-catalog.ts`). The Go implementation uses `time_tracking.view_own`, `time_tracking.view_all`, `time_tracking.edit`, and `booking_overview.delete_bookings`. We will use the existing permissions.
- **Soft-Delete**: The ticket says "Soft-Delete" for the delete mutation, but the Prisma Booking model has NO `deletedAt` field and the Go implementation uses hard delete. Adding soft-delete would require a schema migration out of scope. We will use **hard delete** matching the existing behavior, but delete derived bookings first (matching Go logic).
- **Time Format**: The ticket mentions `time?, end_time?, duration_minutes?` as input fields. The existing model uses `originalTime` and `editedTime` as integer minutes from midnight (0-1439). There is no `end_time` or `duration_minutes`. We will use the existing model fields. The handler will accept HH:MM string input and convert to minutes (matching Go handler pattern).
- **Source Field**: The Go handler hardcodes `BookingSourceWeb` ("web") for web-created bookings. We will do the same, not accepting arbitrary source values from the client.
- **Pair Bookings**: The `pair_id` is a read-only field on bookings. Pairing is handled by the recalculation service (TICKET-235). The CRUD router exposes `pairId` in the output but does NOT manage pairing.
- **Data Scope**: The Go handler applies data scope filtering via `scopeFromContext()` which checks employee-level access. For the tRPC router, we will use `applyDataScope()` middleware and build scope-aware WHERE clauses for the `employee` relation, following the pattern in `employees.ts:427-436`.
- **Derived Bookings**: The Go service creates auto-generated derived bookings when a booking has a `BookingReasonID` with adjustment configuration. This business logic is ported.
- **15 Consumer Files**: The booking hooks are used in 15 files across the frontend. The hook migration must maintain backward-compatible function signatures.

```typescript
function parseTimeString(time: string): number {
  const [hoursStr, minutesStr] = time.split(":")
  const hours = parseInt(hoursStr, 10)
  const minutes = parseInt(minutesStr, 10)
  if (isNaN(hours) || isNaN(minutes) || hours < 0 || hours > 23 || minutes < 0 || minutes > 59) {
    throw new TRPCError({
      code: "BAD_REQUEST",
      message: "Invalid time format, expected HH:MM (00:00 - 23:59)",
    })
  }
  return hours * 60 + minutes
}
```

**Data scope helpers** (adapted from `employees.ts:427-436` for booking context):

```typescript
function buildBookingDataScopeWhere(dataScope: DataScope): Record<string, unknown> | null {
  if (dataScope.type === "department") {
    return { employee: { departmentId: { in: dataScope.departmentIds } } }
  } else if (dataScope.type === "employee") {
    return { employeeId: { in: dataScope.employeeIds } }
  }
  return null
}

function checkBookingDataScope(
  dataScope: DataScope,
  booking: { employeeId: string; employee?: { departmentId: string | null } | null }
): void {
  if (dataScope.type === "department") {
    if (
      !booking.employee?.departmentId ||
      !dataScope.departmentIds.includes(booking.employee.departmentId)
    ) {
      throw new TRPCError({
        code: "FORBIDDEN",
        message: "Booking not within data scope",
      })
    }
  } else if (dataScope.type === "employee") {
    if (!dataScope.employeeIds.includes(booking.employeeId)) {
      throw new TRPCError({
        code: "FORBIDDEN",
        message: "Booking not within data scope",
      })
    }
  }
}
```

### 2.1 Function: `createDerivedBookingIfNeeded`

```typescript
async function createDerivedBookingIfNeeded(
  prisma: PrismaClient,
  original: BookingRecord,
  tenantId: string,
  createdBy: string | null
): Promise<void> {
  if (!original.bookingReasonId) return

  // Load reason with adjustment fields
  const reason = await prisma.bookingReason.findFirst({
    where: { id: original.bookingReasonId, tenantId },
  })
  if (!reason) return
  if (!reason.referenceTime || reason.offsetMinutes === null) return  // No adjustment configured

  // Resolve reference time
  const refMinutes = await resolveReferenceTime(prisma, reason, original)
  if (refMinutes === null) return

  // Calculate derived time = reference + offset, clamped to 0-1439
  let derivedTime = refMinutes + reason.offsetMinutes
  if (derivedTime < 0) derivedTime = 0
  if (derivedTime > MAX_TIME_MINUTES) derivedTime = MAX_TIME_MINUTES

  // Determine derived booking type
  const derivedBookingTypeId = reason.adjustmentBookingTypeId || original.bookingTypeId

  // Idempotent: check if a derived booking already exists
  const existingDerived = await prisma.booking.findFirst({
    where: { originalBookingId: original.id },
  })

  if (existingDerived) {
    // Update existing derived booking
    await prisma.booking.update({
      where: { id: existingDerived.id },
      data: {
        editedTime: derivedTime,
        originalTime: derivedTime,
        bookingTypeId: derivedBookingTypeId,
        calculatedTime: null,
      },
    })
    return
  }

  // Create new derived booking
  await prisma.booking.create({
    data: {
      tenantId: original.tenantId,
      employeeId: original.employeeId,
      bookingDate: original.bookingDate,
      bookingTypeId: derivedBookingTypeId,
      originalTime: derivedTime,
      editedTime: derivedTime,
      source: "derived",
      isAutoGenerated: true,
      originalBookingId: original.id,
      bookingReasonId: original.bookingReasonId,
      notes: `Auto-generated from reason: ${reason.code}`,
      createdBy: createdBy,
      updatedBy: createdBy,
    },
  })
}
```

### 2.2 Function: `resolveReferenceTime`

Port of Go `resolveReferenceTime` (`service/booking.go:372-412`):

```typescript
async function resolveReferenceTime(
  prisma: PrismaClient,
  reason: BookingReasonRecord,
  original: BookingRecord
): Promise<number | null> {
  if (!reason.referenceTime) return null

  switch (reason.referenceTime) {
    case "booking_time":
      return original.editedTime

    case "plan_start": {
      const edp = await prisma.employeeDayPlan.findFirst({
        where: {
          employeeId: original.employeeId,
          planDate: original.bookingDate,
        },
        include: { dayPlan: true },
      })
      if (!edp?.dayPlan?.comeFrom) return original.editedTime  // Fallback
      return edp.dayPlan.comeFrom
    }

    case "plan_end": {
      const edp = await prisma.employeeDayPlan.findFirst({
        where: {
          employeeId: original.employeeId,
          planDate: original.bookingDate,
        },
        include: { dayPlan: true },
      })
      if (!edp?.dayPlan?.goFrom) return original.editedTime  // Fallback
      return edp.dayPlan.goFrom
    }

    default:
      return null
  }
}
```

**New implementation:**

```typescript
import { useTRPC } from "@/trpc"
import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query"

interface UseBookingsOptions {
  employeeId?: string
  from?: string
  to?: string
  pageSize?: number
  page?: number
  enabled?: boolean
}

export function useBookings(options: UseBookingsOptions = {}) {
  const { employeeId, from, to, pageSize = 50, page, enabled = true } = options
  const trpc = useTRPC()
  return useQuery(
    trpc.bookings.list.queryOptions(
      {
        employeeId,
        fromDate: from,
        toDate: to,
        pageSize,
        page,
      },
      { enabled }
    )
  )
}

export function useBooking(id: string, enabled = true) {
  const trpc = useTRPC()
  return useQuery(
    trpc.bookings.getById.queryOptions({ id }, { enabled: enabled && !!id })
  )
}

export function useCreateBooking() {
  const trpc = useTRPC()
  const queryClient = useQueryClient()
  return useMutation({
    ...trpc.bookings.create.mutationOptions(),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: trpc.bookings.list.queryKey() })
      queryClient.invalidateQueries({ queryKey: trpc.bookings.getById.queryKey() })
    },
  })
}

export function useUpdateBooking() {
  const trpc = useTRPC()
  const queryClient = useQueryClient()
  return useMutation({
    ...trpc.bookings.update.mutationOptions(),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: trpc.bookings.list.queryKey() })
      queryClient.invalidateQueries({ queryKey: trpc.bookings.getById.queryKey() })
    },
  })
}

export function useDeleteBooking() {
  const trpc = useTRPC()
  const queryClient = useQueryClient()
  return useMutation({
    ...trpc.bookings.delete.mutationOptions(),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: trpc.bookings.list.queryKey() })
      queryClient.invalidateQueries({ queryKey: trpc.bookings.getById.queryKey() })
    },
  })
}
```

## Phase 5: Consumer Audit and Adaptation

### Purpose

Review all 15 consumer files to ensure they are compatible with the new tRPC hook signatures. The query hooks (`useBookings`, `useBooking`) return data in a slightly different shape (tRPC output vs REST response), and mutation hooks expect different input shapes.

### Key Differences to Check

1. **Query response shape**: Old REST returns `{ data: Booking[], total: number }`. New tRPC returns `{ items: Booking[], total: number }`. Consumers accessing `.data.data` need to change to `.data.items`.

2. **Mutation input shape**: Old REST mutations send a raw body object. New tRPC mutations expect the Zod-validated input shape. For `useCreateBooking`, the input is now `{ employeeId, bookingTypeId, bookingDate, time, notes?, bookingReasonId? }` instead of the old REST `{ employee_id, booking_type_id, ... }` (camelCase vs snake_case).

3. **Booking field names**: The tRPC output uses camelCase (`bookingDate`, `editedTime`, `bookingTypeId`), matching Prisma field names. The old REST API may have used snake_case. Consumers referencing booking fields need to match.
