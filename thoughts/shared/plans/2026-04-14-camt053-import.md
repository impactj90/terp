# CAMT.053 Bankkontoauszug-Import

## Overview

CAMT.053.001.08-XML-Dateien in Terp hochladen, parsen, in Bank-Transaktionen
zerlegen und automatisch gegen offene Ausgangsrechnungen (Credit-Pfad) und
offene Eingangsrechnungen (Debit-Pfad) matchen. Nicht matchende
Buchungen landen in einer neuen Bank-Inbox unter `/finance/bank-inbox`,
in der der Buchhalter manuell zuordnen oder ignorieren kann. Auto-Matches
erzeugen dieselben `BillingPayment` / `InboundInvoicePayment`-Rows wie
die manuellen Pfade heute schon — zusätzlich eine Allocation-Row, die
die Bank-Transaktion mit der Payment-Row verknüpft. Skonto-aware
Betragsvergleich, Platform-Subscription-Filter, Konsistenz-Check für
bereits per `PaymentRun.markBooked` bezahlte Rechnungen.

Feature wird über neues Modul `bank_statements` gegated und fünf neue
Permissions (`bank_transactions.{view,import,match,unmatch,ignore}`).
ADMIN + BUCHHALTUNG voller Zugriff, VORGESETZTER read-only.

Alle Preflight-Items (IBAN-Unique-Constraint, Mahnwesen-Refresh,
`InboundInvoice.paymentStatus` + `InboundInvoicePayment`-Modell) sind
laut Commit-History auf Staging seit `a00d603b` aktiv — dieser Plan
baut auf dieser Basis auf.

## Current State Analysis

Siehe `thoughts/shared/research/2026-04-14-camt053-import-implementation.md`
für die vollständige Bestandsaufnahme. Kurzfassung:

- **XML-Parsing** läuft heute über `fast-xml-parser` v5.5.10 in zwei Rollen
  (`src/lib/services/zugferd-xml-parser.ts:118`, `src/lib/services/payment-run-xml-generator.ts:238`).
  Defensive Walking-Helper in `zugferd-xml-parser.ts:64-113` sind direkt
  als Vorbild für einen CAMT-Parser nutzbar.
- **Polymorphie-Pattern B** (separate Tabellen pro Parent-Typ) ist für
  Zahlungs-Entitäten dominant — `billing_payments`, `inbound_invoice_payments`,
  `wh_supplier_payments`. Die neuen Allocation-Tabellen folgen diesem
  Muster.
- **IBAN-Lookup** auf `CrmBankAccount` existiert heute nicht. Der
  Phase-1-Preflight hat den Composite-Index `(tenant_id, iban)` bereits
  angelegt (`20260425000000_crm_bank_accounts_iban_unique.sql`) — neue
  Query nutzt ihn ab dem ersten Lesen.
- **Matching-Write-APIs** existieren: `billing-payment-service.ts:252-437`
  (`createPayment` / `cancelPayment` mit Skonto-Cascade) und
  `inbound-invoice-payment-service.ts:114-270` (analog, ohne Skonto).
  Der neue Matcher ruft diese Services auf, schreibt nicht direkt.
- **Storage-Bucket-Pattern**: `payment-runs`-Bucket
  (`supabase/migrations/20260423000002_create_payment_runs_storage_bucket.sql`)
  ist der direkte Vorfahr. Drei Upload-Patterns existieren — Pattern A
  (Direct-Base64-tRPC) wie bei inbound-invoices ist das nächstliegende
  Muster für CAMT, weil der Server die Datei synchron parsen muss.
- **Permission-Seed-Template**:
  `supabase/migrations/20260423000001_add_payment_run_permissions_and_module.sql`
  zeigt das `jsonb_agg(DISTINCT val) UNION ALL`-Pattern für idempotentes
  Role-Merging.
- **Inbox-UI-Patterns**: `dunning-runs-tab` + `dunning-reminder-detail-sheet`
  für List→Sheet, `platform/tenants/convert-requests/page.tsx` für Tabbed
  Inbox, `payment-runs/proposal-section` für Multi-Row-Allocation mit
  Sticky Footer und Running-Total.
- **Audit-Logs** sind fire-and-forget via `audit-logs-service.ts:173-213`,
  Impersonation-Dual-Write läuft transparent via AsyncLocalStorage in
  `src/lib/platform/impersonation-context.ts` und `src/trpc/init.ts:308`.
- **Number-Sequence-Prefixes** sind tenant-konfigurierbar
  (`src/lib/services/number-sequence-service.ts:35-63`, Keys: `invoice`,
  `inbound_invoice`, `credit_note`). Matcher muss Prefix-Snapshot pro
  Import-Lauf lesen, nicht auf `RE-` / `ER-` hardcoden.

## Desired End State

**Nach Phase 1**: Schema mit drei neuen Tabellen (`bank_statements`,
`bank_transactions`, plus die zwei Allocation-Tabellen) und neuen
`bankAllocationId`-Spalten auf `BillingPayment` + `InboundInvoicePayment`
ist grün. Prisma-Client generiert ohne Fehler. Bestehende Testsuite
bleibt unverändert grün. Kein Service, kein tRPC-Endpoint, keine UI.

**Nach Phase 2**: Fünf Permissions (`bank_transactions.{view,import,match,unmatch,ignore}`)
sind in `permission-catalog.ts` deklariert und an ADMIN + BUCHHALTUNG +
VORGESETZTER (view only) seeded. Neues Modul `bank_statements` ist im
CHECK-Constraint auf `tenant_modules.module` erlaubt und in
`AVAILABLE_MODULES` gelistet. Neuer privater Storage-Bucket
`bank-statements` (5 MiB, `application/xml` + `text/xml`) existiert auf
staging und lokaler Dev-DB.

**Nach Phase 3**: Reiner Parser `bank-statement-camt-parser.ts` existiert,
nimmt einen XML-String und liefert ein strukturiertes
`ParsedCamtStatement`-Objekt zurück. Error-Klassen `CamtParseError`
(fällt auf `INTERNAL_SERVER_ERROR`) und `CamtValidationError`
(`BAD_REQUEST`) sind deklariert. Unit-Tests mit inline XML-Fixtures
decken sechs Szenarien ab. Kein DB-Write, kein Matcher.

**Nach Phase 4**: Upload-Flow läuft end-to-end ohne Matcher.
`bankStatements.import`-tRPC-Mutation nimmt Base64-XML, berechnet
SHA-256, prüft Dedup, parst, legt `BankStatement` + N `BankTransaction`-
Rows an (alle auf `status=unmatched`), lädt das XML in den Bucket. Ein
Re-Upload derselben Datei liefert idempotent "bereits importiert".
Permission-Gate (`bank_transactions.import`) blockiert fremde User.
Unit- + Integration-Tests abdecken Happy-Path, Dedup, Validation-Error,
Permission-Gate.

**Nach Phase 5**: Credit-Matcher-Pfad ist aktiv. Neue Query
`crmAddressRepository.findAddressByIban(prisma, tenantId, iban)` nutzt
den Preflight-Index. `bank-transaction-matcher-service.ts` mit
Credit-Pfad matcht CREDIT-Transaktionen gegen offene BillingDocument-
Rechnungen, berücksichtigt Skonto-aware Toleranz, Platform-Subscription-
Filter, Verwendungszweck-Regex aus Tenant-Prefixes, und setzt
`suggestedAddressId` auch bei Teilmatches. Bei Eindeutigkeit erzeugt
der Matcher eine `BillingPayment`-Row (via `billingPaymentService.createPayment`)
+ `BillingDocumentBankAllocation`-Row, setzt `BankTransaction.status=matched`.
Audit-Log pro Match.

**Nach Phase 6**: Debit-Matcher-Pfad ist aktiv. DEBIT-Transaktionen
werden gegen offene `InboundInvoice`-Rechnungen gematcht, mit
`invoiceNumber`-Prio über `number`, `sellerIban`-Fallback,
`endToEndId`-Boost aus `PaymentRunItem` und Konsistenz-Check für bereits
per `markBooked` auf PAID gesetzte Rechnungen (Audit-only
`consistency_confirmed`, keine neue Payment-Row). Integration-Tests
decken End-to-End Upload → Match ab (beide Pfade), Atomicity, Audit-Log,
Impersonation-Dual-Write.

**Nach Phase 7**: Inbox-UI unter `/finance/bank-inbox` ist live. Tabbed
Layout (`unmatched | matched | ignored`), Row-Click öffnet Right-Sheet
mit Split-Allocation-UI, vorbelegten Kunden-Filter (via
`suggestedAddressId`), Sticky Footer mit Running-Total, "Match
bestätigen"-Button (enabled wenn Σ allokiert = tx.amount ±0.01) und
"Ignorieren"-Dialog mit optionaler Notiz. Neue Sidebar-Sektion
`financeSection` mit Modul-Gate `bank_statements`. i18n-Keys in beiden
Sprachdateien. Playwright-E2E-Spec `65-bank-inbox.spec.ts` deckt
Happy-Path Credit, Split-Allocation, Ignore, Locale-Switch und
Permission-Gate.

**Nach Phase 8**: Unmatch-Flow komplett. `bankTransactionService.unmatch`
storniert die Payment-Row über `billingPaymentService.cancelPayment` bzw.
`inboundInvoicePaymentService.cancelPayment`, löscht die Allocation-Row,
setzt `BankTransaction.status → unmatched`. `BillingDocument.paymentStatus`
wird live neu berechnet (über den bestehenden Service-Pfad). Eine zweite
Unmatch auf derselben Transaktion schlägt mit klarem Fehler fehl.
Performance-Checks mit einem CAMT-File von ~500 Buchungen (synthetisch).
i18n-Lücken geschlossen.

### Key Discoveries

- **Matcher ruft bestehende Services auf, nicht direkt Prisma** —
  `billing-payment-service.ts` und `inbound-invoice-payment-service.ts`
  tragen bereits alle Guards (Status-Check, Overpayment, Skonto-Cascade,
  Consistency-Check). Der Matcher bleibt "decision-schwer, business-logic-
  leicht".
- **Chicken-and-Egg-Problem BankAllocationId ↔ PaymentRow**: Die
  Allocation-Row braucht die `billingPaymentId` aus `createPayment`, und
  `BillingPayment.bankAllocationId` braucht die `allocationId`. Lösung:
  Create-Reihenfolge `createPayment → createAllocation → update
  BillingPayment.bankAllocationId`. Alle drei in derselben Transaktion.
- **Platform-Subscription-Filter** (`reminder-eligibility-service.ts:181-182`)
  ist das einzige bestehende Konstrukt für "Dokument aus Kandidatenliste
  streichen, wenn Platform-Subscription-Marker im internalNotes steht".
  Matcher reuses diesen Helper.
- **`getApplicableDiscount(doc, paymentDate)`** (`src/lib/billing/payment-discount.ts:1-35`)
  erwartet das Payment-Datum, um die Skonto-Tier-Frist zu prüfen. Der
  Matcher übergibt `tx.valueDate` als Payment-Datum — dies entspricht
  dem Tag, an dem das Geld wirtschaftlich beim Empfänger ankommt.
- **`PaymentRunItem.endToEndId`** existiert bereits (`prisma/schema.prisma:6390`)
  und ist beim XML-Export via `payment-run-xml-generator` gefüllt. Der
  Debit-Matcher kann darüber rückwirkend einen vom eigenen PaymentRun
  ausgelösten Überweisungs-Eintrag in CAMT erkennen.
- **Die Bestands-Testsuite hat 6 pre-existing fails auf staging**
  (permission count, module-pricing, weekPlans, login-service, e2e/01 +
  e2e/04). Jede neue Phase fährt auf diese Baseline, nicht auf "alles
  grün".

## What We're NOT Doing

- **Kein** manueller Reconciliation-Flow für Alt-Zahlungen, die vor
  CAMT-Einführung manuell erfasst wurden.
- **Kein** Dashboard oder KPI-Reporting für Bank-Inbox-Durchsatz
  (wie viele Buchungen wurden diesen Monat automatisch gematcht).
- **Keine** gesonderte UI zur Verwaltung mehrerer Eingangs-Bankkonten
  pro Tenant. Mehrere Statements parallel sind erlaubt (`account_iban`
  ist pro `BankStatement` gespeichert), aber es gibt kein "welches
  Konto ist welches"-Management-UI.
- **Kein** automatischer Import per Bank-API (HBCI, FinTS, EBICS).
  Upload ist manuell durch den Buchhalter.
- **Kein** Export-Pfad — CAMT ist Empfang, nicht Generierung.
- **Kein** separater UI-Pfad für Platform-Subscription-Rechnungen im
  CAMT-Inbox. Diese werden vom Matcher übersprungen (`hasPlatformSubscriptionMarker`-
  Filter) und bleiben dauerhaft `unmatched`. Der Operator bucht sie
  manuell im normalen Billing-UI des Operator-Tenants.
- **Keine** Fremdwährungs-Auto-Matching. Parsing + Speichern läuft,
  aber `currency !== "EUR"` → `status=unmatched` mit expliziter
  Inbox-Meldung. Keine Wechselkurs-Logik.
- **Keine** Bankgebühr-Pauschale im Matching. B2B-DACH-Überweisungen
  haben praktisch keine abgezogenen Gebühren — die Skonto-aware
  Toleranz ist der einzige nicht-exakte Match-Pfad.
- **Keine** XSD-Validierung. Parsing ist format-tolerant mit
  Pflichtfeld-Checks im Service-Exit (konsistent mit ZUGFeRD-Parser-
  Muster).
- **Keine** zeitgebundene Undo-Lane für Auto-Matches (kein "24h-Undo").
  Cancel-Pfad ist jederzeit möglich über den normalen Payment-Cancel-
  Pfad.
- **Keine** Prisma-`@relation`-Deklarationen zwischen Platform-Modellen
  und den neuen Bank-Tabellen. Bank-Tabellen sind reine Terp-
  Domain und haben keine Platform-Kopplung.
- **Keine** Generalisierung bestehender Payment-Services. `billingPaymentService`
  und `inboundInvoicePaymentService` werden nicht refactored — der Matcher
  ruft sie als Client auf.

## Implementation Approach

Acht Phasen strikt sequentiell mit manueller Verifikations-Pause nach
jeder Phase. Jede Phase liefert einen eigenen PR und wird einzeln auf
staging deployt. Phasen 1 + 2 können technisch parallel entwickelt
werden, aber der Deploy erfolgt strikt sequentiell, damit Review-
Kollisionen ausgeschlossen sind. Phase 3 ist komplett unabhängig vom
Schema-Stand, wird aber erst nach Phase 1 gemerged, damit die
Error-Klassen und das Parser-Output-Shape zu den Prisma-Typen passen.

Pro Phase im "Success Criteria"-Block wird explizit aufgelistet, welche
Test-Dateien und welche i18n-Keys neu dazukommen. Der Block teilt sich
strikt in **Automated Verification** (Commands, die automatisch
ausgeführt werden können) und **Manual Verification** (UI-/Locale-/
Browser-Checks durch Tolga). Jede UI-Phase hat einen expliziten
"Sprache einmal auf EN umschalten, keine Raw-Keys sichtbar"-Punkt.

## Phase 1 — Schema + Modelle

### Overview

Drei neue Tabellen (`bank_statements`, `bank_transactions`, zwei
Allocation-Tabellen), Enums, Indexe, Unique-Constraints, zwei neue
`bank_allocation_id`-Spalten auf `billing_payments` und
`inbound_invoice_payments`. Prisma-Schema-Updates spiegeln die SQL-
Tabellen 1:1. Kein Service, kein Router, keine UI.

### Changes Required

#### 1. Neue Supabase-Migration — Bank-Statement + Bank-Transaction

**File**: `supabase/migrations/20260427000000_create_bank_statements_and_transactions.sql`

**Changes**:

```sql
-- =============================================================
-- CAMT.053 Phase 1: BankStatement + BankTransaction + Enums
-- Plan: thoughts/shared/plans/2026-04-14-camt053-import.md
-- =============================================================

-- Enums
CREATE TYPE bank_transaction_direction AS ENUM ('CREDIT', 'DEBIT');
CREATE TYPE bank_transaction_status    AS ENUM ('unmatched', 'matched', 'ignored');

-- BankStatement — Kopf-Tabelle pro hochgeladener CAMT-Datei
CREATE TABLE bank_statements (
  id                UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id         UUID        NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  file_name         VARCHAR(255) NOT NULL,
  sha256_hash       CHAR(64)    NOT NULL,
  xml_storage_path  TEXT        NOT NULL,
  account_iban      VARCHAR(34) NOT NULL,
  statement_id      VARCHAR(255) NOT NULL,
  period_from       TIMESTAMPTZ NOT NULL,
  period_to         TIMESTAMPTZ NOT NULL,
  opening_balance   DOUBLE PRECISION NOT NULL,
  closing_balance   DOUBLE PRECISION NOT NULL,
  currency          CHAR(3)     NOT NULL,
  imported_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  imported_by_id    UUID,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE UNIQUE INDEX uq_bank_statements_tenant_hash
  ON bank_statements(tenant_id, sha256_hash);
CREATE INDEX idx_bank_statements_tenant_imported_at
  ON bank_statements(tenant_id, imported_at DESC);

CREATE TRIGGER set_bank_statements_updated_at
  BEFORE UPDATE ON bank_statements
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

-- BankTransaction — eine Row pro CAMT-Buchung (iteriert über
-- Stmt/Ntry/NtryDtls/TxDtls, nicht pro <Ntry>-Atom).
CREATE TABLE bank_transactions (
  id                   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id            UUID        NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  statement_id         UUID        NOT NULL REFERENCES bank_statements(id) ON DELETE CASCADE,
  booking_date         TIMESTAMPTZ NOT NULL,
  value_date           TIMESTAMPTZ NOT NULL,
  amount               DOUBLE PRECISION NOT NULL,
  currency             CHAR(3)     NOT NULL,
  direction            bank_transaction_direction NOT NULL,
  counterparty_iban    VARCHAR(34),
  counterparty_name    TEXT,
  counterparty_bic     VARCHAR(11),
  remittance_info      TEXT,
  end_to_end_id        VARCHAR(255),
  mandate_id           VARCHAR(255),
  bank_reference       VARCHAR(255),
  bank_tx_code         JSONB,
  status               bank_transaction_status NOT NULL DEFAULT 'unmatched',
  suggested_address_id UUID REFERENCES crm_addresses(id) ON DELETE SET NULL,
  ignored_at           TIMESTAMPTZ,
  ignored_by_id        UUID,
  ignored_reason       TEXT,
  created_at           TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at           TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_bank_transactions_tenant_status
  ON bank_transactions(tenant_id, status);
CREATE INDEX idx_bank_transactions_tenant_booking_date
  ON bank_transactions(tenant_id, booking_date);
CREATE INDEX idx_bank_transactions_tenant_counterparty_iban
  ON bank_transactions(tenant_id, counterparty_iban);
CREATE INDEX idx_bank_transactions_tenant_statement
  ON bank_transactions(tenant_id, statement_id);

CREATE TRIGGER set_bank_transactions_updated_at
  BEFORE UPDATE ON bank_transactions
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

**Implementation Notes**:
- `sha256_hash` ist `CHAR(64)` für hex-encoded SHA-256. Unique-Constraint
  greift auf `(tenant_id, sha256_hash)` für Datei-Dedup beim Upload.
- `statement_id` ist aus `<Stmt>/<Id>` und kann pro Tenant über die Zeit
  wiederkehren (Sparkasse nummeriert manche Statements zyklisch) — daher
  keine Unique-Constraint drauf.
- `currency` als `CHAR(3)` (ISO 4217). `counterparty_iban` als
  `VARCHAR(34)` (IBAN-Max-Länge).
- `bank_tx_code` als JSONB, weil der CAMT-Code ein 4-stufiges Tupel
  (`Domn`/`Fmly`/`SubFmlyCd`) ist und wir kein 3-Spalten-Blähen wollen.
- `suggested_address_id` ist nullable, FK mit `ON DELETE SET NULL` —
  der Match-Vorschlag verfällt wenn der Kunde gelöscht wird, die
  BankTransaction bleibt erhalten.
- Kein `tenant_id`-REFERENCES-Check auf der Allocation-Tabelle zu
  CrmAddress — Cross-Tenant-Konsistenz wird im Service-Code sichergestellt.

#### 2. Neue Supabase-Migration — Allocation-Tabellen + Rück-FK-Spalten

**File**: `supabase/migrations/20260427000001_create_bank_transaction_allocations.sql`

**Changes**:

```sql
-- =============================================================
-- CAMT.053 Phase 1: Zwei Allocation-Tabellen + Rück-FK-Spalten
-- Plan: thoughts/shared/plans/2026-04-14-camt053-import.md
--
-- Pattern B (zwei separate Tabellen pro Parent-Typ), konsistent mit
-- billing_payments vs. inbound_invoice_payments.
-- =============================================================

-- Credit-Pfad: BillingDocument-Allocation
CREATE TABLE billing_document_bank_allocations (
  id                   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id            UUID        NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  bank_transaction_id  UUID        NOT NULL REFERENCES bank_transactions(id) ON DELETE CASCADE,
  billing_document_id  UUID        NOT NULL REFERENCES billing_documents(id) ON DELETE RESTRICT,
  billing_payment_id   UUID        REFERENCES billing_payments(id) ON DELETE RESTRICT,
  amount               DOUBLE PRECISION NOT NULL,
  auto_matched         BOOLEAN     NOT NULL DEFAULT FALSE,
  matched_at           TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  matched_by_id        UUID,
  created_at           TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_billing_doc_bank_alloc_tenant_tx
  ON billing_document_bank_allocations(tenant_id, bank_transaction_id);
CREATE INDEX idx_billing_doc_bank_alloc_tenant_doc
  ON billing_document_bank_allocations(tenant_id, billing_document_id);
CREATE UNIQUE INDEX uq_billing_doc_bank_alloc_payment
  ON billing_document_bank_allocations(billing_payment_id)
  WHERE billing_payment_id IS NOT NULL;

-- Debit-Pfad: InboundInvoice-Allocation
CREATE TABLE inbound_invoice_bank_allocations (
  id                        UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id                 UUID        NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  bank_transaction_id       UUID        NOT NULL REFERENCES bank_transactions(id) ON DELETE CASCADE,
  inbound_invoice_id        UUID        NOT NULL REFERENCES inbound_invoices(id) ON DELETE RESTRICT,
  inbound_invoice_payment_id UUID       REFERENCES inbound_invoice_payments(id) ON DELETE RESTRICT,
  amount                    DOUBLE PRECISION NOT NULL,
  auto_matched              BOOLEAN     NOT NULL DEFAULT FALSE,
  matched_at                TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  matched_by_id             UUID,
  created_at                TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_inbound_inv_bank_alloc_tenant_tx
  ON inbound_invoice_bank_allocations(tenant_id, bank_transaction_id);
CREATE INDEX idx_inbound_inv_bank_alloc_tenant_inv
  ON inbound_invoice_bank_allocations(tenant_id, inbound_invoice_id);
CREATE UNIQUE INDEX uq_inbound_inv_bank_alloc_payment
  ON inbound_invoice_bank_allocations(inbound_invoice_payment_id)
  WHERE inbound_invoice_payment_id IS NOT NULL;

-- Rück-FK-Spalten auf den Payment-Tabellen. Keine FK-Enforcement an
-- dieser Stelle — die Verbindung läuft vom Allocation-Record aus.
ALTER TABLE billing_payments
  ADD COLUMN bank_allocation_id UUID;
ALTER TABLE inbound_invoice_payments
  ADD COLUMN bank_allocation_id UUID;
```

**Implementation Notes**:
- `billing_payment_id` / `inbound_invoice_payment_id` sind nullable, weil
  eine Allocation-Row kurz vor der Payment-Creation existieren kann
  (oder bei Consistency-Matches ganz ohne Payment-Row — siehe Phase 6).
  Das partial Unique-Index (`WHERE ... IS NOT NULL`) verhindert, dass
  zwei Allocation-Rows dieselbe PaymentRow referenzieren.
- `billing_document_id` / `inbound_invoice_id` sind NOT NULL — eine
  Allocation ohne Rechnungs-Zuordnung ergibt keinen Sinn.
- `ON DELETE RESTRICT` auf den Rechnungs- und Payment-FKs: das Löschen
  einer bereits gebuchten Rechnung wird blockiert, wenn eine
  Bank-Allocation darauf zeigt. Das ist strenger als `SET NULL`, aber
  konsistent mit `inbound_invoice_payments.invoice_id`.
- `bank_allocation_id` auf `billing_payments` / `inbound_invoice_payments`
  hat **keine** FK-Enforcement — der Rück-Link wird im Service-Code
  gesetzt, nachdem die Allocation-Row existiert (chicken-and-egg
  auflösen). Ein harter FK würde einen Prisma-Relation-Loop produzieren,
  der die Generierung verkomplizieren würde.
- Keine `CREATE TRIGGER set_..._updated_at` — Allocation-Rows sind
  unveränderlich nach dem Insert (kein `updated_at`).

Interne Struktur:

```ts
import { XMLParser } from "fast-xml-parser"

const parser = new XMLParser({
  ignoreAttributes: false,
  attributeNamePrefix: "@_",
  removeNSPrefix: true, // CAMT-Namespaces sind variabel (001.02 vs 001.08) — wir akzeptieren beides
  isArray: (name) => name === "Stmt" || name === "Ntry" || name === "TxDtls" || name === "Bal",
})

function str(val: unknown): string | null { /* wie zugferd */ }
function num(val: unknown): number { /* wie zugferd */ }
function parseCamtDate(val: unknown): Date | null {
  // <Dt>2026-04-10</Dt>  → Date, ISO-8601 short form
  // <DtTm>2026-04-10T14:23:00Z</DtTm> → Date, full form
}
function ensureArray<T>(val: T | T[] | undefined): T[] { /* wie zugferd */ }

export function parseCamt053(xml: string): ParsedCamtStatement {
  let raw: unknown
  try {
    raw = parser.parse(xml)
  } catch (err) {
    throw new CamtValidationError(
      `CAMT-XML konnte nicht geparst werden: ${(err as Error).message}`
    )
  }

  const doc = (raw as Record<string, any>)?.Document
  if (!doc) {
    throw new CamtValidationError("Kein <Document>-Root-Element. Datei ist kein CAMT.053.")
  }
  const bkToCstmr = doc.BkToCstmrStmt
  if (!bkToCstmr) {
    throw new CamtValidationError("Kein <BkToCstmrStmt>-Element. Datei ist kein CAMT.053.")
  }

  const statements = ensureArray(bkToCstmr.Stmt)
  if (statements.length === 0) {
    throw new CamtValidationError("Keine <Stmt>-Elemente gefunden.")
  }
  if (statements.length > 1) {
    // Für Phase 3 eng interpretiert: Ein File = ein Statement.
    // Multi-Statement-Files sind selten; Bank-Exports sind üblicherweise 1:1.
    throw new CamtValidationError(
      `Multi-Statement-Files werden nicht unterstützt (${statements.length} <Stmt>). Bitte pro Statement einzeln importieren.`
    )
  }

  const stmt = statements[0]
  const statementId = str(stmt.Id)
  if (!statementId) {
    throw new CamtValidationError("statement id missing (<Stmt>/<Id>)")
  }

  const accountIban = str(stmt?.Acct?.Id?.IBAN)
  if (!accountIban) {
    throw new CamtValidationError("account IBAN missing (<Stmt>/<Acct>/<Id>/<IBAN>)")
  }

  const currency = str(stmt?.Acct?.Ccy) ?? "EUR"
  // Period aus <FrToDt>/<FrDtTm> + <ToDtTm>
  const periodFrom = parseCamtDate(stmt?.FrToDt?.FrDtTm)
  const periodTo = parseCamtDate(stmt?.FrToDt?.ToDtTm)
  if (!periodFrom || !periodTo) {
    throw new CamtValidationError("statement period missing (<FrToDt>)")
  }

  // Balances: zwei Einträge mit <Tp>/<CdOrPrtry>/<Cd> = OPBD und CLBD
  const balances = ensureArray(stmt.Bal)
  const opening = balances.find((b: any) => str(b?.Tp?.CdOrPrtry?.Cd) === "OPBD")
  const closing = balances.find((b: any) => str(b?.Tp?.CdOrPrtry?.Cd) === "CLBD")
  const openingBalance = opening ? num(opening?.Amt) : 0
  const closingBalance = closing ? num(closing?.Amt) : 0

  // Iteriere immer über Stmt/Ntry/NtryDtls/TxDtls (Batch-aware)
  const transactions: ParsedCamtTransaction[] = []
  for (const ntry of ensureArray(stmt.Ntry)) {
    // PDNG-Einträge (provisorisch) ignorieren. `<Sts>` kann entweder
    // als Objekt (`<Sts><Cd>PDNG</Cd></Sts>`) oder in älteren Schemas
    // als Text-Kind (`<Sts>PDNG</Sts>`) auftauchen — beides abdecken.
    if (str(ntry?.Sts?.Cd) === "PDNG" || str(ntry?.Sts) === "PDNG") continue

    const entryDirection: "CREDIT" | "DEBIT" =
      str(ntry?.CdtDbtInd) === "CRDT" ? "CREDIT" : "DEBIT"
    const entryBookingDate = parseCamtDate(ntry?.BookgDt?.Dt ?? ntry?.BookgDt?.DtTm)
    const entryValueDate = parseCamtDate(ntry?.ValDt?.Dt ?? ntry?.ValDt?.DtTm)
    if (!entryBookingDate || !entryValueDate) {
      throw new CamtValidationError("booking or value date missing on entry")
    }

    const entryAmt = num(ntry?.Amt)
    const entryCurrency = str(ntry?.Amt?.["@_Ccy"]) ?? currency
    const entryBankRef = str(ntry?.AcctSvcrRef)
    const bankTxCode = {
      domain: str(ntry?.BkTxCd?.Domn?.Cd),
      family: str(ntry?.BkTxCd?.Domn?.Fmly?.Cd),
      subFamily: str(ntry?.BkTxCd?.Domn?.Fmly?.SubFmlyCd),
    }

    const ntryDtls = ntry?.NtryDtls
    const txDtlsList = ensureArray(ntryDtls?.TxDtls)
    if (txDtlsList.length === 0) {
      // Ntry ohne TxDtls — fallback auf die Entry-Ebene
      transactions.push({
        bookingDate: entryBookingDate,
        valueDate: entryValueDate,
        amount: entryAmt,
        currency: entryCurrency,
        direction: entryDirection,
        counterpartyIban: null,
        counterpartyName: null,
        counterpartyBic: null,
        remittanceInfo: null,
        endToEndId: null,
        mandateId: null,
        bankReference: entryBankRef,
        bankTxCode,
      })
      continue
    }

    for (const tx of txDtlsList) {
      const counterpartyIban =
        entryDirection === "CREDIT"
          ? str(tx?.RltdPties?.DbtrAcct?.Id?.IBAN)
          : str(tx?.RltdPties?.CdtrAcct?.Id?.IBAN)
      const counterpartyName =
        entryDirection === "CREDIT"
          ? str(tx?.RltdPties?.Dbtr?.Nm) ?? str(tx?.RltdPties?.Dbtr?.Pty?.Nm)
          : str(tx?.RltdPties?.Cdtr?.Nm) ?? str(tx?.RltdPties?.Cdtr?.Pty?.Nm)
      const counterpartyBic =
        entryDirection === "CREDIT"
          ? str(tx?.RltdAgts?.DbtrAgt?.FinInstnId?.BIC) ?? str(tx?.RltdAgts?.DbtrAgt?.FinInstnId?.BICFI)
          : str(tx?.RltdAgts?.CdtrAgt?.FinInstnId?.BIC) ?? str(tx?.RltdAgts?.CdtrAgt?.FinInstnId?.BICFI)

      const ustrdList = ensureArray(tx?.RmtInf?.Ustrd).map(str).filter(Boolean)
      const remittanceInfo = ustrdList.length > 0 ? ustrdList.join("\n") : null
      const endToEndId = str(tx?.Refs?.EndToEndId)
      const mandateId = str(tx?.Refs?.MndtId)
      const txAmt = num(tx?.Amt)
      const txCurrency = str(tx?.Amt?.["@_Ccy"]) ?? entryCurrency

      transactions.push({
        bookingDate: entryBookingDate,
        valueDate: entryValueDate,
        amount: txAmt > 0 ? txAmt : entryAmt, // fallback
        currency: txCurrency,
        direction: entryDirection,
        counterpartyIban,
        counterpartyName,
        counterpartyBic,
        remittanceInfo,
        endToEndId,
        mandateId,
        bankReference: entryBankRef,
        bankTxCode,
      })
    }
  }

  return {
    statementId,
    accountIban,
    currency,
    periodFrom,
    periodTo,
    openingBalance,
    closingBalance,
    transactions,
  }
}
```

**Implementation Notes**:
- `removeNSPrefix: true` akzeptiert `urn:iso:std:iso:20022:tech:xsd:camt.053.001.08`
  und ältere 001.02-Files. Alternativ: `removeNSPrefix: false` und auf
  jeder Ebene doppelte Fallbacks schreiben. Erstere Variante ist kompakter
  und funktioniert, weil CAMT im Ggs. zu ZUGFeRD keine gemischten
  Namespaces innerhalb desselben Dokuments hat.

  Die Auto-Matcher-Filterung passiert erst in Phase 5.
- Bei `<Sts>/<Cd>=PDNG` (provisorisch) wird das gesamte `Ntry` übersprungen —
  nicht in `transactions` geschrieben.
- Multi-Statement-Files werden abgelehnt. Echte Bank-Exports haben ein
  Statement pro Datei.

#### 2. Service — Upload-Flow ohne Matcher

**File**: `src/lib/services/bank-statement-service.ts`

**Changes**: Neue Datei mit der Haupt-Funktion
`importCamtStatement(prisma, tenantId, input, userId)`:

```ts
import crypto from "node:crypto"
import * as storage from "@/lib/supabase/storage"
import { parseCamt053, CamtValidationError } from "./bank-statement-camt-parser"
import * as repo from "./bank-statement-repository"
import * as auditLog from "./audit-logs-service"

export class BankStatementNotFoundError extends Error { /* ... */ }
export class BankStatementValidationError extends Error { /* ... */ }
export class BankStatementConflictError extends Error { /* ... */ }

const BUCKET = "bank-statements"

export interface ImportCamtInput {
  fileBase64: string
  fileName: string
}

export interface ImportCamtResult {
  statementId: string
  alreadyImported: boolean
  transactionsImported: number
  autoMatched: number
  unmatched: number
  ignored: number
}

export async function importCamtStatement(
  prisma: PrismaClient,
  tenantId: string,
  input: ImportCamtInput,
  userId: string | null
): Promise<ImportCamtResult> {
  const buffer = Buffer.from(input.fileBase64, "base64")
  const sha256 = crypto.createHash("sha256").update(buffer).digest("hex")

  // 1. Dedup: wenn Hash bereits vorhanden → idempotenter Return
  const existing = await repo.findStatementByHash(prisma, tenantId, sha256)
  if (existing) {
    return {
      statementId: existing.id,
      alreadyImported: true,
      transactionsImported: 0,
      autoMatched: 0,
      unmatched: 0,
      ignored: 0,
    }
  }

  // 2. Parse (wirft CamtValidationError bei Format-Fehlern)
  const parsed = parseCamt053(buffer.toString("utf8"))

  // 3. Storage-Path vorab festlegen. ID lokal generieren, damit der
  //    Upload vor dem DB-Insert laufen kann.
  const statementRowId = crypto.randomUUID()
  const storagePath = `${tenantId}/${statementRowId}.xml`

  // 4. Storage-Upload AUSSERHALB der Prisma-Transaktion. Storage-I/O
  //    darf keine DB-Connections blockieren. Wenn der DB-Block
  //    anschließend wirft, räumen wir den Bucket im catch-Block
  //    wieder auf.
  await storage.upload(BUCKET, storagePath, buffer, {
    contentType: "application/xml",
    upsert: false,
  })

  try {
    // 5. Atomare DB-Writes
    const result = await prisma.$transaction(async (tx) => {
      const statement = await repo.createStatement(tx, {
        id: statementRowId,
        tenantId,
        fileName: input.fileName,
        sha256Hash: sha256,
        xmlStoragePath: storagePath,
        accountIban: parsed.accountIban,
        statementId: parsed.statementId,
        periodFrom: parsed.periodFrom,
        periodTo: parsed.periodTo,
        openingBalance: parsed.openingBalance,
        closingBalance: parsed.closingBalance,
        currency: parsed.currency,
        importedById: userId,
      })

      const transactionRows = parsed.transactions.map((t) => ({
        tenantId,
        statementId: statement.id,
        bookingDate: t.bookingDate,
        valueDate: t.valueDate,
        amount: t.amount,
        currency: t.currency,
        direction: t.direction,
        counterpartyIban: t.counterpartyIban,
        counterpartyName: t.counterpartyName,
        counterpartyBic: t.counterpartyBic,
        remittanceInfo: t.remittanceInfo,
        endToEndId: t.endToEndId,
        mandateId: t.mandateId,
        bankReference: t.bankReference,
        bankTxCode: t.bankTxCode as any,
        status: "unmatched" as const,
      }))

      await tx.bankTransaction.createMany({ data: transactionRows })

      // Audit-Log
      await auditLog.log(tx, {
        tenantId,
        userId,
        action: "import",
        entityType: "bank_statement",
        entityId: statement.id,
        entityName: input.fileName,
        metadata: {
          transactionsImported: transactionRows.length,
          accountIban: parsed.accountIban,
          statementId: parsed.statementId,
        },
      }).catch(() => {})

      return {
        statementId: statement.id,
        alreadyImported: false,
        transactionsImported: transactionRows.length,
        autoMatched: 0,
        unmatched: transactionRows.length,
        ignored: 0,
      }
    })

    return result
  } catch (err) {
    // Kompensation: Upload war erfolgreich, DB-Transaktion ist
    // gescheitert — Datei im Bucket wieder entfernen, damit kein
    // verwaister Storage-Eintrag zurückbleibt.
    await storage.remove(BUCKET, [storagePath]).catch(() => {})
    throw err
  }
}
```

**Implementation Notes**:
- Storage-Upload läuft **außerhalb** der Prisma-Transaktion. Storage-I/O
  gegen remote Supabase dauert 3-5 Sekunden und darf keine
  DB-Connection blockieren. Stattdessen wird die Statement-ID lokal
  via `crypto.randomUUID()` generiert, der Upload vor dem DB-Block
  ausgeführt, und im `catch`-Block des umklammernden `try` wird der
  Bucket-Eintrag im Fehlerfall per `storage.remove` kompensiert
  (best-effort, swallow).
- `repo.createStatement` muss die vorgegebene ID akzeptieren — das
  Input-Shape bekommt ein optionales `id`-Feld. Ohne vorgegebene ID
  fällt die Implementation auf `gen_random_uuid()`-Default zurück.
- Parallel-Race auf identischen Dateien: zwei User laden dieselbe
  Datei gleichzeitig hoch, beide passieren den Dedup-Check, beide
  uploaden, einer gewinnt den `unique(tenantId, sha256Hash)`-Insert,
  der andere wirft `P2002`. Im `catch` wird der zweite Upload
  kompensiert und der User bekommt eine klare Conflict-Fehlermeldung
  (`BankStatementConflictError`, mapped auf `CONFLICT`). `upsert:
  false` auf dem Storage-Call sorgt dafür, dass der zweite Upload
  nicht den ersten überschreibt — Supabase gibt einen Duplikat-Fehler
  zurück, den wir als `BankStatementConflictError` an den Caller
  weitergeben.
- `bankTransaction.createMany` ist effizienter als eine Schleife, aber
  der Matcher in Phase 5/6 braucht die generierten IDs. Lösung: nach
  `createMany` via `findMany({ where: { statementId } })` re-lesen.
- `xmlStoragePath` wird in zwei Schritten gesetzt (placeholder → update),
  weil wir die statement-ID erst nach dem Insert kennen. Alternativ:
  ID selbst mit `crypto.randomUUID()` generieren und als explizite ID
  an `createStatement` übergeben.

```ts
/**
 * Baut eine Regex, die Rechnungsnummern mit den tenant-konfigurierten
 * Prefixes im Verwendungszweck erkennt.
 */
export function buildInvoiceNumberRegex(snapshot: TenantPrefixSnapshot): RegExp {
  const escape = (s: string) => s.replace(/[.*+?^${}()|[\]\\]/g, "\\$&")
  const prefixes = [snapshot.invoicePrefix, snapshot.creditNotePrefix].map(escape).join("|")
  return new RegExp(`(${prefixes})\\d+`, "gi")
}

export function buildInboundNumberRegex(snapshot: TenantPrefixSnapshot): RegExp {
  return new RegExp(`(${snapshot.inboundInvoicePrefix.replace(/[.*+?^${}()|[\]\\]/g, "\\$&")})\\d+`, "gi")
}

/**
 * Findet alle Rechnungsnummer-Treffer (inkl. Position) in einem
 * Verwendungszweck.
 */
export function extractInvoiceNumbers(remittanceInfo: string | null, regex: RegExp): string[] {
  if (!remittanceInfo) return []
  return Array.from(remittanceInfo.matchAll(regex)).map((m) => m[0])
}

/**
 * Prüft, ob ein BillingDocument mit Skonto-aware Toleranz zum
 * Transaktions-Betrag passt. Returnt den Match-Modus.
 */
export function compareAmountWithSkonto(
  txAmount: number,
  document: {
    totalGross: number
    effectiveTotalGross: number
    discountPercent: number | null
    discountTier1Days?: number
    discountTier2Percent?: number | null
    discountTier2Days?: number
    invoiceDate: Date
  },
  paymentDate: Date,
  getApplicableDiscount: (doc: any, date: Date) => { percent: number; tier: 1 | 2 } | null,
): { match: "exact" | "skonto" | "none"; discount?: { percent: number; tier: 1 | 2 } } {
  const tolerance = 0.01
  if (Math.abs(txAmount - document.effectiveTotalGross) <= tolerance) {
    return { match: "exact" }
  }
  const applicable = getApplicableDiscount(document, paymentDate)
  if (applicable) {
    const expectedAfterDiscount =
      document.effectiveTotalGross * (1 - applicable.percent / 100)
    if (Math.abs(txAmount - expectedAfterDiscount) <= tolerance) {
      return { match: "skonto", discount: applicable }
    }
  }
  return { match: "none" }
}
```

```ts
import * as auditLog from "./audit-logs-service"

export class BankTransactionMatchError extends Error { ... }
export class BankTransactionMatchConflictError extends Error { ... }

export interface MatchDecision {
  result: "matched" | "unmatched"
  reason?: string
  suggestedAddressId?: string | null
  allocation?: {
    billingDocumentId: string
    amount: number
    effectiveTotalGross: number
    discount?: { percent: number; tier: 1 | 2 }
  }
}

/**
 * Pure decision function — wird isoliert getestet.
 */
export async function computeCreditMatchDecision(
  tx: Prisma.TransactionClient,
  tenantId: string,
  bankTx: BankTransaction,
  snapshot: TenantPrefixSnapshot,
): Promise<MatchDecision> {
  // 1. Fremdwährung raus
  if (bankTx.currency !== "EUR") {
    return { result: "unmatched", reason: "foreign_currency" }
  }

  // 2. IBAN-Lookup
  const iban = normalizeIban(bankTx.counterpartyIban ?? "")
  if (!iban) {
    return { result: "unmatched", reason: "no_iban" }
  }
  const addressHit = await crmAddressRepo.findAddressByIban(tx, tenantId, iban)
  if (!addressHit) {
    return { result: "unmatched", reason: "iban_unknown" }
  }

  // 3. Offene Rechnungen für diese Adresse laden
  const openItems = await billingPaymentRepo.findOpenItems(tx, tenantId, {
    addressId: addressHit.addressId,
  })

  // 4. Platform-Subscription-Filter
  const eligible = openItems.items.filter(
    (doc) => !hasPlatformSubscriptionMarker(doc.internalNotes ?? ""),
  )

  if (eligible.length === 0) {
    return { result: "unmatched", reason: "no_open_items", suggestedAddressId: addressHit.addressId }
  }

  // 5. Verwendungszweck-Plausibilität
  const regex = buildInvoiceNumberRegex(snapshot)
  const refs = extractInvoiceNumbers(bankTx.remittanceInfo, regex)

  // 6. Betrag-Eindeutigkeit mit Skonto-aware Toleranz
  const enriched = eligible.map((doc) => billingPaymentService.enrichOpenItem(doc))
  const candidates = enriched.map((doc) => ({
    doc,
    comparison: compareAmountWithSkonto(
      bankTx.amount,
      {
        totalGross: doc.totalGross,
        effectiveTotalGross: doc.effectiveTotalGross,
        discountPercent: doc.discountPercent ?? null,
        invoiceDate: doc.date,
      },
      bankTx.valueDate,
      getApplicableDiscount,
    ),
  }))
  const amountMatches = candidates.filter((c) => c.comparison.match !== "none")

  // 6a. Kein Betrag-Match → nur Suggestion speichern
  if (amountMatches.length === 0) {
    return { result: "unmatched", reason: "no_amount_match", suggestedAddressId: addressHit.addressId }
  }

  // 6b. Mehrere Betrag-Matches → Verwendungszweck als Tiebreaker
  let unique = amountMatches[0]
  if (amountMatches.length > 1) {
    const withRefHit = refs.length > 0
      ? amountMatches.filter((c) => refs.some((r) => r === c.doc.number))
      : []
    if (withRefHit.length === 1) {
      unique = withRefHit[0]
    } else {
      return { result: "unmatched", reason: "ambiguous", suggestedAddressId: addressHit.addressId }
    }
  }

  // 6c. Verwendungszweck-Widerspruch: wenn Regex einen Treffer liefert,
  //     der NICHT die Kandidaten-Nummer ist → Match abbrechen
  if (refs.length > 0 && !refs.includes(unique.doc.number)) {
    return { result: "unmatched", reason: "remittance_conflict", suggestedAddressId: addressHit.addressId }
  }

  return {
    result: "matched",
    suggestedAddressId: addressHit.addressId,
    allocation: {
      billingDocumentId: unique.doc.id,
      amount: bankTx.amount,
      effectiveTotalGross: unique.doc.effectiveTotalGross,
      discount: unique.comparison.discount,
    },
  }
}

/**
 * Side-effect-heavy Wrapper — führt die Decision aus und schreibt.
 * Wird vom Upload-Flow (Phase 5) und vom manuellen Match (Phase 7)
 * aufgerufen.
 */
export async function runCreditMatchForTransaction(
  tx: Prisma.TransactionClient,
  tenantId: string,
  bankTxId: string,
  snapshot: TenantPrefixSnapshot,
  userId: string | null,
): Promise<MatchDecision> {
  const bankTx = await tx.bankTransaction.findUniqueOrThrow({ where: { id: bankTxId } })
  if (bankTx.direction !== "CREDIT") {
    throw new BankTransactionMatchError("runCreditMatch called on non-credit transaction")
  }

  const decision = await computeCreditMatchDecision(tx, tenantId, bankTx, snapshot)

  // suggestedAddressId auch bei unmatched setzen (Teilmatch-Fall)
  if (decision.suggestedAddressId !== undefined) {
    await tx.bankTransaction.update({
      where: { id: bankTxId },
      data: { suggestedAddressId: decision.suggestedAddressId },
    })
  }

  if (decision.result !== "matched" || !decision.allocation) return decision

  // Write-Phase: createPayment → createAllocation → update BankTx.status +
  //              update BillingPayment.bankAllocationId
  const payment = await billingPaymentService.createPayment(tx, tenantId, {
    documentId: decision.allocation.billingDocumentId,
    date: bankTx.valueDate,
    amount: decision.allocation.amount,
    type: "BANK",
    notes: `CAMT ${bankTx.bankReference ?? bankTx.id}`,
    isDiscount: false,
  }, userId)

  // Skonto-Zeile, falls vorhanden. Der Betrag ist die Differenz
  // zwischen dem effektiven Rechnungsbetrag und der tatsächlich
  // überwiesenen Summe — also das, was der Kunde durch die Skonto-
  // Frist gespart hat. Die `isDiscount=true`-Zeile gleicht damit die
  // Rechnung auf PAID aus. Vorbild: bestehender Skonto-Pfad in
  // `billing-payment-service.ts`.
  if (decision.allocation.discount) {
    const discountAmount = round2(
      decision.allocation.effectiveTotalGross - bankTx.amount,
    )
    await billingPaymentService.createPayment(tx, tenantId, {
      documentId: decision.allocation.billingDocumentId,
      date: bankTx.valueDate,
      amount: discountAmount,
      type: "BANK",
      notes: `CAMT Skonto ${decision.allocation.discount.percent}%`,
      isDiscount: true,
    }, userId)
  }

  const allocation = await tx.billingDocumentBankAllocation.create({
    data: {
      tenantId,
      bankTransactionId: bankTxId,
      billingDocumentId: decision.allocation.billingDocumentId,
      billingPaymentId: payment.id,
      amount: decision.allocation.amount,
      autoMatched: true,
      matchedById: userId,
    },
  })

  // Chicken-and-Egg auflösen: BillingPayment.bankAllocationId setzen
  await tx.billingPayment.update({
    where: { id: payment.id },
    data: { bankAllocationId: allocation.id },
  })

  await tx.bankTransaction.update({
    where: { id: bankTxId },
    data: { status: "matched" },
  })

  await auditLog.log(tx, {
    tenantId,
    userId,
    action: "match",
    entityType: "bank_transaction",
    entityId: bankTxId,
    metadata: {
      allocationId: allocation.id,
      billingDocumentId: decision.allocation.billingDocumentId,
      amount: decision.allocation.amount,
      auto: true,
    },
  }).catch(() => {})

  return decision
}
```

## Phase 6 — Matcher-Engine Debit-Pfad

### Overview

Erweiterung von `bank-transaction-matcher-service.ts` um den DEBIT-Pfad.
Primär-Heuristik ist der Verwendungszweck-Regex auf
`InboundInvoice.invoiceNumber` (Lieferanten-Nummer, höhere Priorität)
und `InboundInvoice.number` (Terp-intern `ER-...`, niedrigere Priorität).
Sekundäre Heuristik: IBAN-Match gegen `InboundInvoice.sellerIban`.
`endToEndId`-Boost für eigene `PaymentRunItem`-Exports. Doppel-Buchung-
Schutz für bereits via `markBooked` auf `PAID` gesetzte Rechnungen
(Audit-only `consistency_confirmed`, keine neue Payment-Row).

#### 1. Matcher-Service — Debit-Pfad

```ts
  tx: Prisma.TransactionClient,
```

Pipeline:

1. **Fremdwährung** → `unmatched`, `reason: "foreign_currency"`
2. **endToEndId-Boost**: Suche nach `PaymentRunItem.endToEndId === bankTx.endToEndId`.
   Wenn Treffer → das verknüpfte `InboundInvoice` ist ein **starker
   Kandidat**. Prüfe `invoice.paymentStatus`:
   - `PAID` → **Doppel-Buchung-Schutz**: Konsistenz-Match, kein
     `createPayment`-Call, kein `Allocation`-Create.
     `BankTransaction.status → matched`, Audit-Log
     `action="consistency_confirmed"`, Return.
   - `UNPAID` / `PARTIAL` → Normal-Flow: `createPayment` + `Allocation`,
     wie Credit-Pfad.
3. **Verwendungszweck-Regex** mit beiden Prefixes (`invoiceNumber`-Prio,
   dann `number`):
   ```ts
   const inboundRegex = buildInboundNumberRegex(snapshot)
   const terpRefs = extractInvoiceNumbers(bankTx.remittanceInfo, inboundRegex)
   const freeformRefs = extractFreeformInvoiceNumbers(bankTx.remittanceInfo) // alphanumerisch, ≥4 chars
   ```
   Lade alle offenen InboundInvoices und filtere:
   - Match auf `invoice.invoiceNumber` (Lieferanten-Nr) in `freeformRefs`
   - Match auf `invoice.number` (Terp) in `terpRefs`
4. **IBAN-Fallback** (sekundär): wenn kein Verwendungszweck-Treffer,
   Match gegen `invoice.sellerIban === normalizeIban(bankTx.counterpartyIban)`
5. **Betrag-Eindeutigkeit + Datum-Toleranz ±3 Tage**:
   Kandidat muss `Math.abs(bankTx.amount - invoice.totalGross) <= 0.01`
   und `Math.abs(valueDate - invoice.dueDate) <= 3 days` erfüllen.
6. **Tiebreaker**: Mehrere Kandidaten + `endToEndId`-Hit priorisiert,
   sonst `unmatched`, `suggestedAddressId = invoice.supplierAddressId`.
7. **suggestedAddressId**: wenn ein Lieferant erkannt wird, aber kein
   eindeutiger Rechnungs-Treffer → `suggestedAddressId = supplier.addressId`.

#### 1a. Manual-Match-Service

**File**: `src/lib/services/bank-transaction-matcher-service.ts`

**Changes**: Neue exportierte Funktion `manualMatchTransaction`,
parallel zu `runCreditMatchForTransaction` / `runDebitMatchForTransaction`.

```ts
export class BankTransactionMatchValidationError extends Error {
  constructor(message: string) {
    super(message)
    this.name = "BankTransactionMatchValidationError"
  }
}

export interface ManualMatchAllocation {
  billingDocumentId?: string
  inboundInvoiceId?: string
  amount: number
}

export async function manualMatchTransaction(
  prisma: PrismaClient,
  tenantId: string,
  bankTransactionId: string,
  allocations: ManualMatchAllocation[],
  userId: string | null,
): Promise<void> {
  if (allocations.length === 0) {
    throw new BankTransactionMatchValidationError("at least one allocation required")
  }

  // Jede Allocation muss genau einen Ziel-Typ haben (XOR).
  for (const a of allocations) {
    const hasCredit = !!a.billingDocumentId
    const hasDebit = !!a.inboundInvoiceId
    if (hasCredit === hasDebit) {
      throw new BankTransactionMatchValidationError(
        "allocation must reference exactly one of billingDocumentId or inboundInvoiceId",
      )
    }
    if (a.amount <= 0) {
      throw new BankTransactionMatchValidationError("allocation amount must be positive")
    }
  }

  await prisma.$transaction(async (tx) => {
    const bankTx = await tx.bankTransaction.findUniqueOrThrow({
      where: { id: bankTransactionId },
    })
    if (bankTx.tenantId !== tenantId) {
      throw new BankTransactionMatchValidationError("cross-tenant access")
    }
    if (bankTx.status !== "unmatched") {
      throw new BankTransactionMatchValidationError(
        `transaction ${bankTransactionId} is not unmatched (status=${bankTx.status})`,
      )
    }

    // Server-Validation: Summe der Allocations ≈ Transaktionsbetrag ±0.01
    const sum = round2(allocations.reduce((s, a) => s + a.amount, 0))
    if (Math.abs(sum - bankTx.amount) > 0.01) {
      throw new BankTransactionMatchValidationError(
        `allocation sum ${sum} does not match transaction amount ${bankTx.amount}`,
      )
    }

    // Platform-Subscription-Filter: manuelle Matches auf Subscription-
    // Rechnungen sind nicht erlaubt (konsistent zu "What We're NOT
    // Doing" — Platform-Subscription-Rechnungen werden im Operator-
    // Tenant im normalen Billing-UI gebucht, nicht im Bank-Inbox).
    const creditDocIds = allocations
      .map((a) => a.billingDocumentId)
      .filter((id): id is string => !!id)
    if (creditDocIds.length > 0) {
      const docs = await tx.billingDocument.findMany({
        where: { tenantId, id: { in: creditDocIds } },
        select: { id: true, internalNotes: true },
      })
      for (const doc of docs) {
        if (hasPlatformSubscriptionMarker(doc.internalNotes ?? "")) {
          throw new BankTransactionMatchValidationError(
            `document ${doc.id} is a platform subscription invoice and cannot be matched via the bank inbox`,
          )
        }
      }
    }

    // Allocations schreiben (mixed credit + debit möglich)
    for (const alloc of allocations) {
      if (alloc.billingDocumentId) {
        const payment = await billingPaymentService.createPayment(tx, tenantId, {
          documentId: alloc.billingDocumentId,
          date: bankTx.valueDate,
          amount: alloc.amount,
          type: "BANK",
          notes: `CAMT manual ${bankTx.bankReference ?? bankTx.id}`,
          isDiscount: false,
        }, userId)

        const allocation = await tx.billingDocumentBankAllocation.create({
          data: {
            tenantId,
            bankTransactionId,
            billingDocumentId: alloc.billingDocumentId,
            billingPaymentId: payment.id,
            amount: alloc.amount,
            autoMatched: false,
            matchedById: userId,
          },
        })
        await tx.billingPayment.update({
          where: { id: payment.id },
          data: { bankAllocationId: allocation.id },
        })
      } else if (alloc.inboundInvoiceId) {
        const payment = await inboundInvoicePaymentService.createPayment(tx, tenantId, {
          invoiceId: alloc.inboundInvoiceId,
          date: bankTx.valueDate,
          amount: alloc.amount,
          type: "BANK",
          notes: `CAMT manual ${bankTx.bankReference ?? bankTx.id}`,
        }, userId)

        const allocation = await tx.inboundInvoiceBankAllocation.create({
          data: {
            tenantId,
            bankTransactionId,
            inboundInvoiceId: alloc.inboundInvoiceId,
            inboundInvoicePaymentId: payment.id,
            amount: alloc.amount,
            autoMatched: false,
            matchedById: userId,
          },
        })
        await tx.inboundInvoicePayment.update({
          where: { id: payment.id },
          data: { bankAllocationId: allocation.id },
        })
      }
    }

    await tx.bankTransaction.update({
      where: { id: bankTransactionId },
      data: { status: "matched" },
    })

    await auditLog.log(tx, {
      tenantId,
      userId,
      action: "match",
      entityType: "bank_transaction",
      entityId: bankTransactionId,
      metadata: {
        auto: false,
        allocationCount: allocations.length,
        amountSum: sum,
      },
    }).catch(() => {})
  })
}
```

**Implementation Notes**:
- `BankTransactionMatchValidationError` endet auf `ValidationError`
  und wird damit über den `handleServiceError`-Suffix-Dispatch auf
  `BAD_REQUEST` gemappt. Der UI-Toast im Detail-Sheet mapt die
  Error-Message auf `bankInbox.toast.matchError`.
- **Mixed-Allocations**: eine einzige manuelle Match-Operation darf
  eine Credit-Allocation und eine Debit-Allocation gleichzeitig
  anlegen — z.B. wenn eine Bank-Gutschrift teilweise eine offene
  Rechnung bedient und teilweise eine Rückerstattung an den Lieferanten
  kompensiert. Das ist ein seltener Fall, aber das Schema unterstützt
  ihn und die Write-Logik iteriert einfach über alle `allocations`
  in einer Transaktion.
- **Platform-Subscription-Filter auch hier**: der Operator soll
  Platform-Subscription-Rechnungen nicht versehentlich aus der Bank-
  Inbox heraus matchen können. Der Check greift nur für Credit-
  Allocations (Debit betrifft Eingangsrechnungen, die per Definition
  keine Platform-Subscription sind).
- **Atomicity**: alle Writes laufen in derselben `$transaction`. Bei
  einem Fehler im zweiten `createPayment`-Call wird der erste
  zurückgerollt, das Status-Update auf `matched` bleibt aus.
- Der Router-Handler `manualMatch` ruft diesen Service auf und fängt
  `BankTransactionMatchValidationError` über `handleServiceError`.

## Phase 8 — Unmatch-Flow + Polish

### Overview

Cancel-Pfad für Auto- und Manual-Matches. Service-Funktion
`unmatchBankTransaction` ruft den bestehenden
`billingPaymentService.cancelPayment` bzw.
`inboundInvoicePaymentService.cancelPayment`, löscht die Allocation-Row,
setzt `BankTransaction.status → unmatched` zurück, schreibt Audit.
Button in der Detail-View des `matched`-Tabs. Abschluss-Polish:
Performance-Check mit synthetischem 500-Buchungen-File, i18n-Lücken,
Edge-Cases.

### Changes Required

#### 1. Unmatch-Service

**File**: `src/lib/services/bank-transaction-matcher-service.ts`

**Changes**: Neue exportierte Funktion:

```ts
export async function unmatchBankTransaction(
  prisma: PrismaClient,
  tenantId: string,
  bankTransactionId: string,
  userId: string | null,
): Promise<void> {
  await prisma.$transaction(async (tx) => {
    const bankTx = await tx.bankTransaction.findUniqueOrThrow({
      where: { id: bankTransactionId },
    })
    if (bankTx.status !== "matched") {
      throw new BankTransactionMatchConflictError(
        `Transaction ${bankTransactionId} is not matched (status=${bankTx.status})`
      )
    }

    // Credit-Allocations
    const creditAllocs = await tx.billingDocumentBankAllocation.findMany({
      where: { tenantId, bankTransactionId },
    })
    for (const alloc of creditAllocs) {
      if (alloc.billingPaymentId) {
        await billingPaymentService.cancelPayment(
          tx, tenantId, alloc.billingPaymentId, userId,
        )
      }
      await tx.billingDocumentBankAllocation.delete({ where: { id: alloc.id } })
    }

    // Debit-Allocations
    const debitAllocs = await tx.inboundInvoiceBankAllocation.findMany({
      where: { tenantId, bankTransactionId },
    })
    for (const alloc of debitAllocs) {
      if (alloc.inboundInvoicePaymentId) {
        await inboundInvoicePaymentService.cancelPayment(
          tx, tenantId, alloc.inboundInvoicePaymentId, userId,
        )
      }
      await tx.inboundInvoiceBankAllocation.delete({ where: { id: alloc.id } })
    }

    await tx.bankTransaction.update({
      where: { id: bankTransactionId },
      data: { status: "unmatched" },
    })

    await auditLog.log(tx, {
      tenantId,
      userId,
      action: "unmatch",
      entityType: "bank_transaction",
      entityId: bankTransactionId,
      metadata: {
        creditAllocationsRemoved: creditAllocs.length,
        debitAllocationsRemoved: debitAllocs.length,
      },
    }).catch(() => {})
  })
}
```

**Implementation Notes**:
- Der `cancelPayment`-Service in `billing-payment-service.ts:439-502`
  berechnet `BillingDocument.paymentStatus` bereits live neu — der
  Unmatch-Flow muss das nicht zusätzlich triggern.
- Consistency-Matches (aus Phase 6) haben keine `billingPaymentId` /
  `inboundInvoicePaymentId` auf der Allocation-Row — der Unmatch-Flow
  muss das abfangen: Falls die Allocation eine `consistency_confirmed`-
  Markierung hat (via `metadata.consistency`), wird sie einfach gelöscht,
  ohne `cancelPayment`-Call. Die ursprüngliche Rechnung ist via
  `markBooked` bezahlt und bleibt `PAID`.
  Konkret: **die Consistency-Matches schreiben gar keine Allocation-Row**
  (siehe Phase 6), d.h. der Unmatch-Flow trifft sie nie an — die
  BankTransaction ist `status=matched`, aber ohne Allocation. Der
  Unmatch-Service muss diesen Fall explizit behandeln: wenn
  `creditAllocs.length === 0 && debitAllocs.length === 0` → nur
  `status → unmatched` setzen + Audit schreiben, nichts zu cancellen.

## Performance Considerations

- **Upload-Flow läuft synchron innerhalb einer tRPC-Mutation**. Bei
  synthetischen 500-Buchungen-Files (Phase-8-Perf-Test) sollte die
  Mutation unter 30s bleiben. Reale CAMT-Dateien eines aktiven Tenants
  liegen im Bereich 30-200 Buchungen pro Monat — Performance ist nicht
  kritisch.
- **Matcher-Query-Count**: pro Transaktion ein `crmBankAccount.findFirst`
  (IBAN-Index) + ein `billingPaymentRepo.findOpenItems`. Für eine
  100-Buchungen-Datei sind das ~200 Queries. Die Queries sind klein
  und indexiert — keine Batching-Optimierung nötig.
- **Storage-Upload** liegt innerhalb der Transaktion, blockiert die
  DB-Connection bis der Upload abgeschlossen ist. 5 MiB Max-Size + lokales
  Supabase bedeutet <1s Upload-Dauer. Bei remote-staging-Supabase kann
  das bis zu 3-5s dauern. Wir akzeptieren das — 5 Sekunden Upload-
  Blockade sind bei einer manuellen Buchhalter-Aktion OK.
- **Index-Nutzung**:
  - `idx_bank_transactions_tenant_status` für Tab-Listings in der Inbox
  - `idx_bank_transactions_tenant_counterparty_iban` für IBAN-basiertes
    Kandidaten-Lookup im Detail-View
  - `idx_crm_bank_accounts_tenant_iban` (aus Preflight) für den
    Matcher-IBAN-Lookup
- **Skalierungs-Grenze**: bei >1000 Buchungen pro Datei sollte der
  Upload asynchron werden (Job-Queue). Phase 1-8 adressiert das nicht —
  siehe "What We're NOT Doing".

## Migration Notes

- **Keine Backfill-Migration**: es gibt keine historischen Bank-
  Transaktionen, die in die neuen Tabellen zu übernehmen wären. Staging
  und prod starten mit leeren `bank_statements`- und `bank_transactions`-
  Tabellen.
- **Rollback**: Phase 1 ist rollback-sicher — `DROP TABLE
  bank_statements CASCADE` + `DROP TABLE bank_transactions CASCADE` +
  `ALTER TABLE billing_payments DROP COLUMN bank_allocation_id` +
  `ALTER TABLE inbound_invoice_payments DROP COLUMN bank_allocation_id`.
  Phase 2 ist rollback-sicher über `DELETE FROM storage.buckets WHERE id
  = 'bank-statements'` + `ALTER TABLE tenant_modules ... REMOVE
  'bank_statements' FROM CHECK`. Phase 3-8 sind Code-only und können
  über Git-Revert zurückgerollt werden.
- **Prisma-Migrations-Konvention**: alle neuen Migrations tragen
  Timestamp `20260427xxxxx` / `20260428xxxxx`. Die Reihenfolge ist:
  1. `20260427000000_create_bank_statements_and_transactions.sql`
  2. `20260427000001_create_bank_transaction_allocations.sql`
  3. `20260428000000_add_bank_transaction_permissions_and_module.sql`
  4. `20260428000001_create_bank_statements_storage_bucket.sql`

  nicht in den neuen Matcher/Service-Files auftauchen außer als

  Pfade werden nicht berührt.
