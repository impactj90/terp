# Plan: Add Missing Database Indexes

## Date: 2026-03-21
## Research: thoughts/shared/research/2026-03-21-missing-database-indexes.md

## Overview

Add two composite database indexes to improve query performance on the two highest-traffic tables:

1. **Booking** table: `(tenant_id, employee_id, booking_date)` -- covers 7 query patterns in daily-calc, bookings-repository, employees-repository, and evaluations-repository. The daily calculation engine hits this pattern for every employee/day combination.
2. **MonthlyValue** table: `(tenant_id, year, month)` -- covers 6 query patterns in monthly-values-repository, payroll-export-repository, and monthly-calc. The monthly evaluation list and payroll export are the primary consumers.

Both indexes supplement (not replace) existing single-column and two-column indexes.

**File:** `prisma/schema.prisma`

## Phase 2: Create Migration

### Steps

1. Create the migration file:
   ```bash
   supabase migration new add_composite_indexes_bookings_monthly_values
   ```
   This will create `supabase/migrations/20260101000108_add_composite_indexes_bookings_monthly_values.sql` (the timestamp prefix and sequence number are auto-generated by Supabase CLI based on the project's convention).

   **Note:** If the Supabase CLI generates a different timestamp format, rename the file to match the project convention: `20260101000108_add_composite_indexes_bookings_monthly_values.sql`.

2. Write the following SQL content to the migration file:

```sql
-- Add composite indexes for high-traffic query patterns
--
-- bookings: covers WHERE tenant_id = X AND employee_id = Y AND booking_date = Z
-- (daily calculation engine, booking lookups, evaluations)
--
-- monthly_values: covers WHERE tenant_id = X AND year = Y AND month = Z
-- (monthly evaluation list, payroll export)

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_bookings_tenant_employee_date
  ON bookings (tenant_id, employee_id, booking_date);

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_mv_tenant_year_month
  ON monthly_values (tenant_id, year, month);
```

**Important notes on the SQL:**
- `CONCURRENTLY` allows index creation without locking writes on production. However, Supabase migrations run inside a transaction by default, and `CREATE INDEX CONCURRENTLY` cannot run inside a transaction. If this is a concern for production deployment, the migration should be split or the transaction wrapper removed. For local dev and staging (small datasets), non-concurrent `CREATE INDEX` is fine. Use the non-concurrent version if the migration runner wraps in a transaction:

```sql
-- Add composite indexes for high-traffic query patterns
--
-- bookings: covers WHERE tenant_id = X AND employee_id = Y AND booking_date = Z
-- (daily calculation engine, booking lookups, evaluations)
--
-- monthly_values: covers WHERE tenant_id = X AND year = Y AND month = Z
-- (monthly evaluation list, payroll export)

CREATE INDEX IF NOT EXISTS idx_bookings_tenant_employee_date
  ON bookings (tenant_id, employee_id, booking_date);

CREATE INDEX IF NOT EXISTS idx_mv_tenant_year_month
  ON monthly_values (tenant_id, year, month);
```

Use the non-concurrent version (second block) since Supabase migrations run in transactions.

## Phase 3: Apply Migration & Regenerate

### Steps

1. **Apply the migration** (reset local DB to apply all migrations including the new one):
   ```bash
   pnpm db:reset
   ```
   Or if you want to apply just the new migration without resetting:
   ```bash
   supabase db push --local
   ```

2. **Regenerate the Prisma client** (picks up the new @@index directives):
   ```bash
   pnpm db:generate
   ```
   This runs `prisma generate` which regenerates the Prisma client types. While indexes don't affect the TypeScript types, it ensures the Prisma schema is valid and parseable.

## Risk Assessment

- **Low risk**: Adding indexes is a non-destructive, additive operation. No schema changes to columns, no data migration.
- **No application code changes required**: Indexes are transparent to Prisma queries.
- **Existing indexes retained**: The old `idx_bookings_employee_date` on `(employee_id, booking_date)` and `idx_mv_tenant` on `(tenant_id)` are kept. PostgreSQL's query planner will automatically prefer the new composite indexes when they provide better selectivity.
- **Storage overhead**: Minimal. Both tables are moderate size, and B-tree indexes on UUID + int/date columns are compact.
